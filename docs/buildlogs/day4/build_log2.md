# Module: Persistent SeenStore Integration and Resume State Investigation

**Date:** July 22, 2026

**Duration:** 2 Hours

## Goal

Implement a persistent visited URL manager that could rebuild the crawler's visited state after application restart and investigate whether it was sufficient to support consistent crawl resumption.

---

## Problem

The crawler maintained a visited URL set only in memory during execution. After restarting the application, all visited information was lost, causing previously downloaded pages to be fetched again.

To solve this, I planned to rebuild the visited set from the crawler's persistent storage instead of introducing a separate persistence file.

While designing this mechanism, another question emerged: **Would rebuilding only the visited URLs be enough for the crawler to resume correctly after a restart?**

---

## What I Tried

### Reused Existing Storage Metadata

Instead of creating a dedicated `seen.txt` file, decided to reuse the existing `index.txt` generated by `PageStorage`.

Since every successfully downloaded page is already recorded inside the storage index, it could also serve as the source for reconstructing the visited URL set.

---

### Implemented SeenStore Initialization

Implemented `initializeStorage()` inside `SeenStore`.

During startup, the class now:

- Opens `index.txt`.
- Reads every stored record.
- Extracts the corresponding URL.
- Inserts the URL into an in-memory hash map representing the visited set.

This ensured that all previously downloaded pages were immediately recognized as visited after restarting the crawler.

---

### Integrated SeenStore with the Crawler

Updated the crawler so that before fetching any URL, it checks whether that URL already exists inside `SeenStore`.

Only URLs that had not been visited previously were allowed to proceed through the fetching pipeline, reducing unnecessary network requests and duplicate page storage.

---

### Investigated Resume Behaviour

Performed a walkthrough of the crawler execution after introducing the persistent visited set.

During the analysis, discovered an important limitation in the current resume strategy.

Although previously downloaded pages were correctly recognized as visited after restarting, the crawler no longer revisited those pages to discover their outgoing links. Since the Frontier was maintained only in memory and was lost after application termination, restarting from the seed URL caused the traversal to stop whenever immediate child pages had already been marked as visited.

This revealed that restoring only the visited set is insufficient for implementing a fully consistent resume mechanism.

---

## Outcome

Successfully implemented a persistent `SeenStore` that reconstructs the crawler's visited URLs from `index.txt` during startup.

However, while validating the resume workflow, identified a significant architectural limitation: restoring visited URLs alone cannot reproduce the crawler's previous traversal state because the Frontier is not persisted. This observation shifted the focus of subsequent development toward designing a more reliable resume strategy.
